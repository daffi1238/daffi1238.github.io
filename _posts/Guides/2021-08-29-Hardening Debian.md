Type=Application
Terminal=false

### Detecting change
One you have a system (in production or not) installed and configured most system files should be relatively static until an upgrade.
At first we can just cehck the whole packages installed with the repositories
`dpkg --verify`
```text
-V, --verify [package-name...]
          Verifies  the integrity of package-name or all packages if omit‐
          ted, by comparing information from the installed paths with  the
          database metadata.
```
dpkg relies on the checksums stored in /var/lib/dpkg/info/package.md5sums so an attacker could modify this file to avoid this check or even compromise the package in your Debian mirror. 
To create a new md5sum file for some modifications to the applications (configurations files) you can launch (For ssh as example):
`/etc/systemd/system/ssh.service override`

To protect correctly against this kind of attack the best option is use an APT digital signature verification system.

Monitoring file with 
**AIDE**
[Manual](https://linux.die.net/man/5/aide.conf)
Advance Intrusion Detection Environment check file integrity save that information in a database (/var/lib/aide/aide.db) saving there information as fingerprints, permissions, timestamps and so on.

Install and initialize
```
sudo apt install aide
#init database
aideinit
# This will execute daily /etc/cron/.daily/aide script to check if there is a change, these will be register in /var/log/aide/*.log and will send to the administrator e-mail defined
```

And attacker that compromise a machine could modify or even delete the local data base, for that is important store the database file on a read-only storage media.

**Tripwire**
This software add the possibility to sign the configuration files so the attacker cannot make it point to a different versions of the reference database

**Samhain**
Similar options but additional functionalities to detect rootkits. This can be launch in a whole network and recolect information in a central server
